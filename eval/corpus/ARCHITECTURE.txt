PROJECT OVERVIEW FundForge is a full-stack crowdfunding platform. It allows: - CREATORS to launch funding campaigns, set goals, add rewards, post updates - DONORS to browse campaigns, donate money, select rewards, leave comments - ADMINS to manage users, moderate content, view platform analytics The system is split into two completely separate applications: backend/ → A REST API server (Node.js) frontend/ → A Single Page Application (React) They communicate exclusively over HTTP using JSON. The backend does not know the frontend exists. The frontend does not care how the backend is implemented internally. This separation is intentional and critical — it means we can replace either side without touching the other. --- ## 2. THE BIG PICTURE — HOW IT ALL CONNECTS When a user opens the app in their browser, here is exactly what happens: 1. Browser requests http://localhost:5173 2. Vite (dev server) serves index.html + compiled React JS 3. React boots up, reads localStorage for saved auth token 4. React renders the UI based on current route (URL) 5. Components call hooks (useApi.ts) which trigger HTTP requests 6. Requests hit http://localhost:4000/api/v1/... 7. Express receives the request, runs middleware chain 8. Controller calls Service, Service calls Repository 9. Repository queries PostgreSQL via Prisma 10. Data flows back: DB → Repository → Service → Controller → JSON response 11. React receives JSON, updates state, re-renders UI Every single user interaction follows this exact flow. Understanding it deeply means you can debug anything. --- ## 3. TECHNOLOGY STACK — EVERY CHOICE EXPLAINED ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ BACKEND ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ NODE.JS What: JavaScript runtime that runs on the server (not in the browser) Why: Same language as frontend = engineers can work both sides Non-blocking I/O = handles thousands of concurrent requests efficiently Massive npm ecosystem = libraries for everything When: Running at all times as our API server process Where: backend/src/index.ts is the entry point TYPESCRIPT What: JavaScript with static type checking Why: Catches bugs at compile time, not runtime. When you write user.emal instead of user.email, TypeScript tells you immediately instead of it failing silently in production at 3am. Enables IDE autocomplete which makes development 3x faster. Self-documenting code — types ARE the documentation. When: Compiled to plain JavaScript before running. Never runs directly. Where: Every .ts file. tsconfig.json defines compiler rules. Note: We use strict mode. This means no implicit 'any' types. If you are tempted to write 'as any' — stop. Fix the type instead. EXPRESS.JS What: Minimal web framework for Node.js Why: Battle-tested, enormous community, extremely flexible. Does not impose structure — we impose our own (layered architecture). Middleware system is elegant and composable. When: Handles every incoming HTTP request Where: backend/src/app.ts sets up the Express application backend/src/index.ts starts the HTTP server How: Request comes in → Express matches route → runs middleware chain → calls route handler → sends response PRISMA ORM What: Type-safe database toolkit for Node.js Why: Writing raw SQL is error-prone and tedious. Prisma generates TypeScript types from your database schema automatically. This means if your database has a 'users' table, Prisma gives you prisma.user.findMany() with full autocomplete and type safety. Migrations are version-controlled — every schema change is tracked. When: Any time we read or write to the database Where: backend/prisma/schema.prisma defines the entire database schema backend/src/config/database.ts exports the Prisma client singleton backend/src/repositories/*.ts use Prisma for all queries Critical: We NEVER write raw SQL. Always use Prisma. This prevents SQL injection attacks and keeps queries type-safe. POSTGRESQL What: The actual database that stores all data Why: Relational database = perfect for our interconnected data model (users have campaigns, campaigns have donations, donations have payments) ACID compliant = transactions are safe (critical for payments) Most battle-tested open source database in existence Supports JSON columns (used for socialLinks, gatewayResponse) When: Prisma connects to it on server startup, queries on every request Where: Runs as a separate process, connection via DATABASE_URL in .env Note: You never interact with PostgreSQL directly in application code. Always go through Prisma. ZOD What: Runtime schema validation library Why: TypeScript types are COMPILE TIME only. At runtime, when a user sends a request, the data could be anything. Zod validates that the actual runtime data matches what we expect. Example: TypeScript says goalAmount is a number. But a user could send goalAmount: "hello". Zod catches this before it reaches the database. When: Runs on every incoming request in our validation middleware Where: backend/src/dtos/*.ts define all Zod schemas backend/src/middleware/validate.middleware.ts applies them How: Request body → Zod schema → if invalid, return 400 error immediately If valid, typed data passed to controller BCRYPT What: Password hashing algorithm Why: NEVER store plaintext passwords. Ever. Bcrypt is a one-way hash designed specifically for passwords. It is intentionally slow (to resist brute-force attacks) and includes a salt (to prevent rainbow table attacks). When: On register (hashing) and login (comparing) Where: backend/src/services/auth.service.ts Note: We use 12 rounds (BCRYPT_SALT_ROUNDS in .env). Higher = slower but more secure. 12 is the industry standard balance. JWT (JSON Web Tokens) What: Stateless authentication tokens Why: Traditional sessions require server-side storage (memory or DB). JWTs are self-contained — the token itself contains the user's ID and roles, signed with a secret key. No database lookup needed to authenticate a request. When: Every authenticated API request includes a JWT in the Authorization header. Our auth middleware verifies it. Where: backend/src/utils/jwt.ts handles signing and verification backend/src/middleware/auth.middleware.ts verifies on each request Two tokens: ACCESS TOKEN - Short-lived (15 minutes). Used for API requests. REFRESH TOKEN - Long-lived (7 days). Used only to get new access tokens. Why two? If an access token is stolen, it expires in 15 min max. The refresh token lives in our database so we can revoke it. HELMET What: Sets secure HTTP headers automatically Why: Browsers have many security features that are opt-in via headers. Helmet sets Content-Security-Policy, X-Frame-Options, X-Content-Type-Options and more — preventing XSS, clickjacking, MIME sniffing attacks without any effort. When: Applied to every response via middleware Where: backend/src/app.ts — app.use(helmet()) MORGAN What: HTTP request logger Why: In development you need to see every request and response code. In production you need logs for debugging production issues. When: Logs every request automatically Where: backend/src/app.ts — only active in non-test environments EXPRESS-RATE-LIMIT What: Limits how many requests an IP can make in a time window Why: Prevents brute-force attacks (trying thousands of passwords), DoS attacks, and API abuse. When: Applied to all /api routes Where: backend/src/app.ts — 100 requests per 15 minutes per IP Note: Returns 429 Too Many Requests when limit is exceeded ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ FRONTEND ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ REACT 18 What: JavaScript library for building user interfaces Why: Component-based = reusable pieces of UI. Declarative = describe what the UI should look like, React figures out how to update it. Massive ecosystem. Virtual DOM = efficient updates. When: Runs entirely in the user's browser after the initial page load Where: frontend/src/main.tsx bootstraps React frontend/src/App.tsx is the root component VITE What: Build tool and development server Why: Extremely fast. Uses native ES modules in development (no bundling = instant hot module replacement). In production, bundles and optimizes everything. Much faster than the old Create React App / Webpack approach. When: Development: runs a dev server with HMR (changes appear instantly) Production: bundles everything into static files in dist/ Where: frontend/vite.config.ts configures it Note: The proxy config in vite.config.ts forwards /api requests to the backend, avoiding CORS issues in development. TAILWINDCSS What: Utility-first CSS framework Why: Write styles directly in JSX using class names. No context switching between CSS files and components. No naming things (no .card-container-wrapper nonsense). The final CSS bundle only includes classes actually used. When: Compiled at build time — Tailwind scans all files for class names Where: frontend/tailwind.config.js configures it frontend/postcss.config.js processes it frontend/src/index.css imports Tailwind directives Critical: Tailwind class names must be complete strings. Never do `text-${color}-500` — Tailwind cannot find dynamic classes. Always write the full class: text-red-500, text-blue-500. REACT ROUTER DOM v6 What: Client-side routing library Why: Single Page Applications need to handle URL changes without full page reloads. React Router maps URLs to components. When: Every time the user navigates (clicks a link or back button) Where: frontend/src/App.tsx defines all routes frontend/src/components/layout/AppShell.tsx has ProtectedRoute Key concepts: <Routes> = Container for all routes <Route> = Maps a URL path to a component <Outlet> = Where nested routes render their content <Link> = Navigation without page reload (use instead of <a>) useNavigate() = Programmatic navigation (after form submission etc.) useParams() = Read URL parameters (:id, :slug) REACT QUERY (TanStack Query) What: Server state management library Why: Fetching data is complex: loading states, error states, caching, refetching, pagination, optimistic updates. React Query handles ALL of this automatically. Without it, every developer writes their own messy useEffect+useState fetching logic differently. When: Any time a component needs data from the API Where: frontend/src/hooks/useApi.ts — all query/mutation hooks defined here frontend/src/App.tsx — QueryClientProvider wraps the entire app Key concepts: useQuery = Fetching data (GET requests). Auto-caches, auto-refetches. useMutation = Modifying data (POST/PATCH/DELETE). Returns loading state. queryKey = Cache key. ['campaigns', params] means "cache this data under this key". Same key = same cache entry. invalidateQueries = "This data is stale, refetch it". Called after mutations to keep UI in sync with server. staleTime = How long cached data is considered fresh (no refetch) ZUSTAND What: Minimal client state management library Why: React's built-in useState is great for local state. But auth state (current user, token) needs to be accessible from ANY component. Redux is too complex for our needs. Zustand is simple, small, and has built-in persist middleware for localStorage. When: Reading or updating auth state from any component Where: frontend/src/store/auth.store.ts What we store: - Current user object (id, email, firstName, lastName, roles) - Access token - Refresh token - isAuthenticated boolean What we DON'T store in Zustand: - Server data (campaigns, donations) — that's React Query's job - Component-local state (modal open/closed) — that's useState's job AXIOS What: HTTP client library Why: Cleaner API than fetch(), automatic JSON parsing, request/response interceptors (critical for our token refresh logic), timeout support, better error handling. When: Every API call from the frontend Where: frontend/src/services/api.ts sets up the configured instance Critical feature — Interceptors: REQUEST interceptor: Automatically attaches the JWT access token to every request header. You never manually add Authorization headers. RESPONSE interceptor: If a request returns 401 (token expired), automatically calls /auth/refresh to get a new token, then retries the original request. The user never knows this happened. If refresh also fails, logs the user out and redirects to /login. RECHARTS What: Chart library built for React Why: Clean API, good default styling, responsive, built with SVG. When: Dashboard analytics pages Where: frontend/src/pages/DashboardPages.tsx and AdminPages.tsx REACT-HOT-TOAST What: Toast notification library Why: Users need feedback when actions succeed or fail. Toasts are non-intrusive, auto-dismiss, and stack properly. When: After mutations (create campaign, donate, delete user etc.) Where: frontend/src/App.tsx — <Toaster> component renders toasts Called anywhere with: toast.success('Done!') or toast.error('Failed') DATE-FNS What: Date utility library Why: JavaScript's built-in Date is painful. date-fns provides clean functions like formatDistanceToNow(date) → "3 minutes ago" When: Formatting dates in the UI (comments, donations, campaign dates) Where: frontend/src/components/campaigns/CommentsSection.tsx --- ## 4. BACKEND DEEP DIVE backend/ ├── prisma/ │ ├── schema.prisma ← THE source of truth for database structure │ └── seed.ts ← Populates DB with initial data for development ├── src/ │ ├── index.ts ← Server entry point. Starts Express, connects DB. │ ├── app.ts ← Express app setup. Registers all middleware. │ ├── config/ │ │ ├── env.ts ← Environment variable validation (Zod) │ │ └── database.ts ← Prisma client singleton │ ├── types/ │ │ └── errors.ts ← ApiError class + HttpStatus enum │ ├── utils/ │ │ ├── jwt.ts ← Token signing and verification │ │ ├── response.ts ← ResponseBuilder utility + pagination meta │ │ ├── slug.ts ← URL slug generation from titles │ │ └── pagination.ts ← Query param parsing helpers │ ├── middleware/ │ │ ├── auth.middleware.ts ← JWT verification, RBAC │ │ ├── validate.middleware.ts ← Zod schema validation │ │ └── error.middleware.ts ← Centralized error handling │ ├── dtos/ │ │ ├── auth.dto.ts ← Register/Login validation schemas │ │ ├── campaign.dto.ts ← Campaign CRUD validation schemas │ │ ├── donation.dto.ts ← Donation validation schemas │ │ └── entities.dto.ts ← All other entity schemas │ ├── repositories/ ← DATA ACCESS LAYER │ ├── services/ ← BUSINESS LOGIC LAYER │ ├── controllers/ ← HTTP HANDLER LAYER │ └── routes/ │ └── index.ts ← All route definitions in one place ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ THE LAYERED ARCHITECTURE — THIS IS THE MOST IMPORTANT CONCEPT ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ We enforce a strict 4-layer architecture. Each layer has ONE job and talks only to the layer directly below it. NEVER skip layers. LAYER 1: ROUTES (routes/index.ts) Job: Map HTTP verbs + paths to controllers. Apply middleware. Rules: No logic here. Only: path → middleware chain → controller method. Example: router.post('/campaigns', authenticate, authorize('CREATOR'), validate(createCampaignDto), CampaignController.create); LAYER 2: CONTROLLERS (controllers/*.ts) Job: Handle HTTP concerns only. Extract data from request. Call service. Return response. Rules: THIN. No business logic. No database access. Should be 5-10 lines per method maximum. Example: async create(req, res, next) { try { const campaign = await service.create(req.body, req.user.sub); return ResponseBuilder.created(res, campaign); } catch (err) { next(err); } } LAYER 3: SERVICES (services/*.ts) Job: Business logic. Validation of business rules. Orchestration. Rules: No HTTP knowledge (no req/res). Throws ApiError for business violations. Can call multiple repositories. Can call other services. Example business logic: - "A campaign can only be activated by an admin" - "A reward cannot be claimed if maxClaims is reached" - "A donation triggers updates to 4 different tables (in a transaction)" This is WHERE the rules of the application live. LAYER 4: REPOSITORIES (repositories/*.ts) Job: Database access only. Raw Prisma queries. Rules: No business logic. Just data access patterns. Methods named clearly: findById, findAll, create, update, softDelete. Returns Prisma model objects. Example: async findById(id: string) { return prisma.campaign.findFirst({ where: { id, deletedAt: null } }); } WHY THIS MATTERS: If you need to change how campaigns are stored → edit the repository only. If you need to change campaign business rules → edit the service only. If you need to change the API response format → edit the controller only. Changes are isolated. Nothing bleeds into other layers. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ MIDDLEWARE — THE REQUEST PIPELINE ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Every request passes through middleware in order before reaching a controller. Think of middleware as a pipeline of filters. Request arrives ↓ helmet() → Set security headers ↓ cors() → Allow requests from our frontend domain only ↓ rateLimit() → Block if too many requests from this IP ↓ express.json() → Parse request body from JSON string to JS object ↓ morgan() → Log the request ↓ authenticate() → Verify JWT token (if route requires auth) ↓ authorize() → Check user has required role (if route requires role) ↓ validate() → Validate request body against Zod schema ↓ Controller → Handle the business logic ↓ errorHandler() → Catches any thrown errors, returns consistent error JSON auth.middleware.ts — THREE exported functions: authenticate → Required auth. Returns 401 if no valid token. authorize(roles)→ Required roles. Returns 403 if user lacks the role. optionalAuth → Attaches user to request if token present, but does not fail if no token. Used for public routes that show extra info to logged-in users. validate.middleware.ts Takes a Zod schema and a request part ('body', 'query', 'params'). Validates the data. If invalid → 400 with detailed field errors. If valid → replaces req[part] with the parsed, typed, safe data. This is why controllers can trust req.body is valid — Zod already checked it. error.middleware.ts THE MOST IMPORTANT MIDDLEWARE. Sits at the very end. Every try/catch in controllers ends with: next(err) This middleware catches ALL errors and returns consistent JSON: { success: false, message: "...", data: null } Handles: ApiError → Uses our error's statusCode and message Prisma errors → Maps database errors to HTTP errors (P2002 unique violation → 409 Conflict) (P2025 not found → 404) Unknown errors→ 500 Internal Server Error ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ DTOs — DATA TRANSFER OBJECTS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ DTOs define the shape of data crossing the API boundary (incoming requests). We use Zod to define them, giving us both validation AND TypeScript types. Example from campaign.dto.ts: export const createCampaignDto = z.object({ title: z.string().min(5).max(200), goalAmount: z.number().positive(), ... }); export type CreateCampaignDto = z.infer<typeof createCampaignDto>; The validate middleware runs createCampaignDto.safeParse(req.body). If it fails → 400 with which fields failed and why. If it passes → req.body is now typed as CreateCampaignDto. Rule: EVERY route that accepts data MUST have a DTO and use validate middleware. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ CONFIG SYSTEM ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ backend/src/config/env.ts validates ALL environment variables at startup. If any required variable is missing or wrong type → server refuses to start. This means you CANNOT deploy with a broken config. It fails loudly. All config accessed via: import { config } from './config/env' NEVER use process.env.VARIABLE directly in application code. Always go through the config object — it is typed and validated. backend/src/config/database.ts exports a singleton Prisma client. Singleton = one database connection pool shared across the entire app. Creating a new PrismaClient for every request would exhaust connections. --- ## 6. Rules: - One file per "page group" (DashboardPages.tsx has all dashboard pages) - Export named functions for each page - Import and compose feature components ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ THE API SERVICE LAYER (services/api.ts) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ This is one of the most important files in the frontend. It creates a configured Axios instance with: baseURL: http://localhost:4000/api/v1 timeout: 15 seconds default headers: Content-Type: application/json REQUEST INTERCEPTOR: Before every request, reads accessToken from localStorage. If found, adds: Authorization: Bearer <token> This happens AUTOMATICALLY. You never manually add auth headers. RESPONSE INTERCEPTOR (the clever part): If response is 401 (Unauthorized): 1. Pause the original request 2. Call POST /auth/refresh with our refresh token 3. If refresh succeeds: save new tokens, retry original request 4. If refresh fails: clear all tokens, redirect to /login 5. If another request also gets 401 during refresh: queue it (do not spam refresh endpoint), retry it when refresh completes This is called "transparent token refresh". Users never see "please log in again" unless their refresh token genuinely expires. The http object exports typed wrappers: http.get<T>(url, params) → GET request, returns ApiResponse<T> http.post<T>(url, body) → POST request http.patch<T>(url, body) → PATCH request http.delete(url) → DELETE request ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ HOOKS (hooks/useApi.ts) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Every API interaction goes through a hook in useApi.ts. NEVER call http.get() or axios directly from a component. Always use or create a hook. WHY? 1. Caching — React Query caches responses. Two components using useCampaigns() share the same cached data. No double fetching. 2. Loading states — isLoading, isError come for free 3. Consistency — all data fetching works the same way 4. Refetching — React Query auto-refetches stale data 5. Invalidation — after a mutation, we invalidate related queries so all components showing that data automatically update QUERY hooks (useQuery) — for fetching: useCampaigns(params) → GET /campaigns useCampaign(id) → GET /campaigns/:id useCampaignBySlug(slug) → GET /campaigns/slug/:slug useUsers(params) → GET /users (admin) useAdminStats() → GET /analytics/admin ... 20+ more MUTATION hooks (useMutation) — for modifying: useCreateCampaign() → POST /campaigns useUpdateCampaign(id) → PATCH /campaigns/:id useDeleteCampaign() → DELETE /campaigns/:id useCreateDonation() → POST /donations ... 15+ more Mutation hooks follow this pattern: 1. mutateAsync(data) called from component 2. HTTP request sent 3. On success → invalidateQueries(['campaigns']) (tells React Query: cached campaign data is stale, refetch) 4. All components using campaign data automatically re-render with fresh data --- ## 7. DATABASE DESIGN PHILOSOPHY ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ NORMALIZATION ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Our database is fully normalized (3rd Normal Form). This means: - No duplicated data - Every piece of data lives in exactly one place - Related data is connected via foreign keys Example: A donation has a campaignId (foreign key to campaigns table). We do NOT store the campaign title on the donation. To get it: JOIN donations with campaigns. Prisma does this via include: {}. WHY? If the campaign title changes, we update ONE row in campaigns. If we stored it on every donation, we would need to update hundreds of rows and they could get out of sync. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ SOFT DELETES ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Most entities have a deletedAt: DateTime? column. When we "delete" a campaign, we set deletedAt = NOW(). The row stays in the database. We never truly delete it. WHY? 1. Data integrity — donations reference campaigns. If we hard-deleted a campaign with donations, those donations would have broken references. 2. Audit trail — know what was deleted and when 3. Recovery — can restore accidentally deleted content 4. Analytics — historical data stays intact All queries filter: WHERE deletedAt IS NULL This is why every findAll query has: where: { deletedAt: null } ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ TRANSACTIONS ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ When creating a donation, we must: 1. Create donation record 2. Create payment record 3. Update campaign raisedAmount 4. Update campaign donationsCount 5. Update donor totalDonated 6. Update reward claimsCount If step 4 fails, steps 1-3 already happened. The data is corrupted. A TRANSACTION means: either ALL steps succeed, or NONE do. If anything fails, the database rolls back to before step 1. In Prisma: prisma.$transaction(async (tx) => { ... }) Inside the callback, use tx instead of prisma for all operations. Rule: Any operation that touches multiple tables MUST use a transaction. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ INDEXES ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Indexes make queries fast. Without an index, the database scans EVERY row. With an index, it jumps directly to matching rows. We index: - Foreign keys (userId, campaignId, donorId) — used in JOINs constantly - Lookup fields (email — used in login) - Filter fields (status, deletedAt — used in WHERE clauses) - Unique fields (slug, email) — also enforce uniqueness Defined in schema.prisma with @@index([fieldName]) --- ## 8. AUTHENTICATION SYSTEM — HOW IT ACTUALLY WORKS Let us trace a complete login flow: STEP 1: User submits login form Frontend: LoginPage calls useAuthStore.login(email, password) STEP 2: Zustand calls API auth.store.ts: http.post('/auth/login', { email, password }) STEP 3: Request hits backend Route: POST /api/v1/auth/login Middleware: validate(loginDto) — checks email format, password not empty Controller: AuthController.login(req, res, next) STEP 4: Service validates credentials AuthService.login(): 1. findUserByEmail(email) — find user in DB 2. If not found → throw ApiError.unauthorized('Invalid credentials') NOTE: We say "Invalid credentials" not "User not found" WHY: Telling attackers the email does not exist helps them 3. bcrypt.compare(password, user.passwordHash) 4. If wrong password → same error message 5. Get user's roles from userRoles join STEP 5: Generate tokens generateTokenPair({ sub: user.id, email, roles }) ACCESS TOKEN: JWT signed with JWT_ACCESS_SECRET, expires 15min Payload: { sub: "uuid", email: "...", roles: ["ADMIN"] } REFRESH TOKEN: JWT signed with JWT_REFRESH_SECRET, expires 7 days Payload: { sub: "uuid" } (minimal, just identifies user) Save refresh token to database (refresh_tokens table) WHY save to DB? So we can revoke it. If token is stolen, delete from DB. STEP 6: Return tokens + user to frontend Frontend: Saves accessToken + refreshToken to localStorage Saves user object to Zustand store Redirects to /dashboard STEP 7: Subsequent API requests Every request: Axios interceptor reads accessToken from localStorage Adds header: Authorization: Bearer eyJ... Backend: authenticate middleware calls verifyAccessToken(token) JWT library verifies signature + expiry If valid: attaches decoded payload to req.user req.user = { sub: "uuid", email: "...", roles: ["ADMIN"] } Controllers access req.user.sub for the current user's ID STEP 8: Token expires (after 15 minutes) API returns 401 Axios response interceptor catches it Calls POST /auth/refresh with refreshToken Backend: verifies refresh token signature + checks DB (not revoked, not expired) If valid: revokes OLD refresh token, creates NEW refresh token (rotation) Returns new access + refresh tokens Axios: saves new tokens, retries original request User notices nothing REFRESH TOKEN ROTATION SECURITY: Every time you use a refresh token, it is immediately invalidated and replaced with a new one. If someone steals your refresh token and tries to use it after you already used it — the system sees a "token reuse" and REVOKES ALL your tokens, forcing a full re-login. This detects token theft automatically. --- ## 9. API DESIGN PRINCIPLES CONSISTENT RESPONSE FORMAT: Every response, success or error, follows this shape: { success: boolean, ← true for success, false for errors data: T | null, ← the actual data (null for errors) message: string, ← human readable description meta?: PaginationMeta ← only on paginated list responses } WHY? Frontend can always do response.data to get the data. response.success to check if it worked. Predictable. Simple. REST CONVENTIONS: GET /campaigns → list (paginated) GET /campaigns/:id → single item POST /campaigns → create PATCH /campaigns/:id → partial update (not PUT = full replace) DELETE /campaigns/:id → delete (soft) WHY PATCH not PUT? PUT requires sending the COMPLETE object. PATCH sends only the fields you want to change. More efficient, less error-prone. NESTED RESOURCES: Resources belonging to another resource are nested in the URL: GET /campaigns/:campaignId/comments → comments for a campaign POST /campaigns/:campaignId/rewards → create reward for a campaign This makes the relationship explicit in the URL. PAGINATION: All list endpoints support: ?page=1&limit=20&sortBy=createdAt&sortOrder=desc Response includes meta: { page, limit, total, totalPages, hasNext, hasPrev } Max limit is 100 — prevents "give me all 10,000 records" attacks HTTP STATUS CODES (we use them correctly): 200 OK → Successful GET/PATCH 201 Created → Successful POST 204 No Content → Successful DELETE (no body) 400 Bad Request → Validation failed 401 Unauthorized → Not logged in (no token or invalid token) 403 Forbidden → Logged in but wrong role 404 Not Found → Resource does not exist 409 Conflict → Duplicate (email already registered) 429 Too Many Requests → Rate limited 500 Internal Error → Something we did not anticipate broke --- ## 10. STATE MANAGEMENT We have TWO kinds of state and they are managed separately: SERVER STATE (data that lives on the server) Examples: list of campaigns, user's donations, analytics stats Managed by: React Query (useQuery + useMutation) Characteristics: Can go stale. Needs refetching. Shared across components. CLIENT STATE (UI state that lives only in the browser) Examples: is the modal open?, what tab is selected?, current user identity Managed by: useState (local) or Zustand (global) Characteristics: Never stale. No server involved. RULE: Never store server data in Zustand. Never use React Query for UI state. They solve different problems. Zustand (auth.store.ts) stores ONLY: - user (current user object) - accessToken - refreshToken - isAuthenticated Everything else (campaigns, donations, users list) is React Query's territory. --- ## 11. ERROR HANDLING STRATEGY Backend — Three-layer error handling: LAYER 1: ApiError class (types/errors.ts) Our custom error class. Extends Error. Has: statusCode, message, isOperational, errors[] Static helpers: ApiError.notFound('Campaign'), ApiError.forbidden() Thrown from services when business rules are violated. LAYER 2: try/catch in controllers Every controller method wraps in try/catch. On catch: next(err) — passes error to Express error middleware. This pattern ensures NO unhandled promise rejection. LAYER 3: errorHandler middleware (error.middleware.ts) Catches everything next(err) sends. Maps to consistent JSON response. Handles ApiError, Prisma errors, unknown errors. In development: includes stack trace. In production: omits it. Frontend — Error handling: React Query: isError + error state on every query hook. Mutations: wrapped in try/catch, toast.error() on failure. Axios interceptor: handles 401 globally (token refresh or logout). No unhandled promise rejections — every async call is caught. --- ## 12. SECURITY MODEL AUTHENTICATION: "Are you who you say you are?" → JWT tokens AUTHORIZATION: "Are you allowed to do this?" → Role-based access control ROLES: ADMIN → Can do everything. User management, content moderation, approve campaigns, access all analytics. CREATOR → Can create/edit/delete OWN campaigns, post updates, manage rewards. Cannot see other creators' analytics. CRUD Create, Read, Update, Delete. The four basic operations on data. DTO Data Transfer Object. Defines the shape of data crossing a boundary. Our DTOs are Zod schemas that validate incoming request data. JWT JSON Web Token. A signed token containing claims (user ID, roles). Cannot be forged without the secret key. MIDDLEWARE A function that runs between receiving a request and handling it. Like an assembly line — each step processes the request. MIGRATION A versioned database schema change. Tracked in git. Running migrations applies changes to the database. ORM Object-Relational Mapper. Lets you query a database using JavaScript/TypeScript instead of writing SQL. PAGINATION Breaking large lists into pages. page=1&limit=20 gives first 20 items. RBAC Role-Based Access Control. Users have roles. Roles grant permissions. REST Representational State Transfer. An API design convention using HTTP methods (GET, POST, PATCH, DELETE) and URLs to represent resources. SEED Populating a fresh database with test/demo data for development. SINGLETON A class that only creates ONE instance ever. Our Prisma client is a singleton — one database connection pool for the whole app. SOFT DELETE Marking a record as deleted (via deletedAt) rather than removing it. The data stays in the database but is filtered from all queries. SPA Single Page Application. The entire frontend is one HTML page. React handles navigation without full page reloads. TRANSACTION A group of database operations that all succeed or all fail together. Critical for operations touching multiple tables (like donations). HOT RELOAD Vite's ability to apply code changes instantly without full page refresh. You save a file, the browser updates in under 100ms. --- This document was written to be exhaustive, not brief.
